# Artifacts

This document describes the v0 artifact direction for reportage.

Artifacts are evidence produced by a reportage run. They allow humans, CI systems, adapters, and later post-processing tools to inspect what happened during E2E execution.

## Canonical record

The artifact bundle written to `.reportage/runs/<run-id>/` — `result.json` plus the raw evidence files it references — is the canonical record of a `reportage run` execution.

`result.json` at the run root is the bundle's canonical manifest. It records:

- run-level outcome (`status`, `processExitCode`, `noop`)
- summary counts
- concrete case results (`tests[]`)
- action results, including evidence references for captured stdout/stderr
- assertion / expectation results
- diagnostics and failure classification
- document-local ids (`test-1`, `action-1`, `assertion-1`, `diagnostic-1`, ...)
- schema version and tool version

`reportage run --format=json` does not print this document. It prints a stdout-safe projection derived from it; see [Relationship to `--format=json`](#relationship-to---formatjson).

## Default behavior

Artifacts are generated by default for every run.

There is currently no `--no-artifacts` mode in the v0 direction. Artifact generation is treated as a normal part of reportage execution, not as an optional reporting feature.

## Contract and validation

The contract for `result.json` is defined in [`spec/artifacts/run-result/schema.json`](../../spec/artifacts/run-result/schema.json), the human- and tool-facing source of truth. See [`spec/artifacts/run-result/README.md`](../../spec/artifacts/run-result/README.md) for the full validation policy.

CI validation is typed Rust deserialization ([`crates/reportage-cli/tests/run_result_fixtures.rs`](../../crates/reportage-cli/tests/run_result_fixtures.rs)): each representative fixture run's `result.json` is deserialised into Rust structs marked `#[serde(deny_unknown_fields)]`, covering the full stable contract the schema defines — every expectation kind, observation enum, and diagnostic shape — not only the shapes the fixtures happen to exercise. An external JSON Schema validator is not required.

### Generated / checked / handwritten boundary

This document mixes three kinds of content, with different staleness guarantees:

- **Checked sections.** The JSON examples below marked with a `<!-- checked-against: ... -->` comment are verified byte-for-byte against fixture snapshots by `run_result_fixtures.rs::docs_artifacts_examples_match_their_fixture_snapshots`; they cannot silently drift from real output. Field lists, required/optional fields, and enums are checked by the typed validation structs against the schema-defined contract, and evidence references (layout, sizes, digests) are checked by `run_result_fixtures.rs::evidence_files_match_their_manifest_references` and [`e2e/artifacts/run-result-manifest.repor`](../../e2e/artifacts/run-result-manifest.repor).
- **Handwritten sections.** The purpose of artifacts, the canonical-record positioning, the relationship to `--format=json`, the raw evidence policy, and the compatibility/stability policy are prose. Tests do not detect drift in this prose; it is maintained by review, with rationale recorded in the ADRs listed at the end of this document.
- **Not generated.** No section of this document is currently machine-generated; "generated or checked" is satisfied by the checks above.

## Stability

The artifact layout and JSON schema should be considered experimental during early v0 implementation.

The pre-#102 `result.json` (snake_case fields, inline base64 stream envelopes, `result: pass|fail|script_error`) was an early v0 experimental contract. Issue #102 replaced it with the canonical schema described here as a deliberate breaking change; early v0 artifact contracts may continue to change this way until stabilization.

Expected stabilization direction:

```text
v0.0.x: canonical schema exists, still experimental
v0.1.x: begin stabilizing the run-result schema
v0.2.x: consider external post-processing and testography integration needs
```

## Default layout

The default artifact root is:

```text
.reportage/
  runs/
    <run-id>/
```

A run layout:

```text
.reportage/
  runs/
    <run-id>/
      result.json
      test-1/
        action-1/
          stdout.bin
          stderr.bin
```

`test-<n>/action-<m>/` segments use the same document-local ids as `result.json`'s `tests[].id` / `tests[].actions[].id`, so an `artifactRef` and the file it names always agree. This correspondence is enforced by the evidence-integrity test and e2e coverage listed in the boundary section above.

Failure classification is carried by `diagnostics[]` (`category`: `parse` / `semantic` / `runtime` / `assertion` / `internal`, plus stable `code` values from [Semantic and assertion diagnostics](semantic-diagnostics.md) and [Parse diagnostics](diagnostics.md)), referenced from failing assertions via `diagnosticRef`. The representative failing, parse-error, semantic-error, runtime-error, and partial-execution shapes are pinned by the snapshots in [`tests/fixtures/run_result/`](../../tests/fixtures/run_result/).

For an empty, whitespace-only, or otherwise valid zero-case suite, the run manifest records a no-op success:

<!-- checked-against: `tests/fixtures/run_result/noop.snapshot.json` -->
```json
{
  "diagnostics": [],
  "noop": true,
  "processExitCode": 0,
  "schemaVersion": 1,
  "status": "passed",
  "summary": {
    "actions": 0,
    "assertions": 0,
    "errors": 0,
    "failed": 0,
    "passed": 0,
    "scripts": 0
  },
  "tests": [],
  "tool": {
    "name": "reportage",
    "version": "<VERSION>"
  }
}
```

No-op execution does not create action or evidence artifacts because no command ran and no assertion was evaluated. The run-level `result.json` is still written because it records that nothing was executed.

## Raw evidence policy

Raw byte evidence — captured stdout/stderr of actions — is never embedded in `result.json` as inline data (base64 or otherwise).

Each action's captured stream is written as a separate raw byte file inside the bundle, and `result.json` references it:

```json
{
  "stdout": {
    "artifactRef": "test-1/action-1/stdout.bin",
    "sizeBytes": 123456,
    "sha256": "..."
  }
}
```

- `artifactRef`: path relative to the directory containing `result.json`
- `sizeBytes`: byte size of the evidence file
- `sha256`: lowercase hex SHA-256 digest of the evidence file's bytes

`sha256` is required: it makes it verifiable that the evidence file inside the bundle is the one the manifest describes.

This keeps the manifest small and streaming-safe regardless of action output size, while the bundle as a whole remains a complete record. Raw byte semantics for stdout/stderr are defined in [semantics.md — stdout/stderr evidence representation](semantics.md#stdoutstderr-evidence-representation).

## Relationship to `--format=json`

`reportage run --format=json` prints a stdout-safe projection of the canonical run result document, defined by its own contract [`spec/output/json-report/schema.json`](../../spec/output/json-report/schema.json) (#89), which this issue's change did not redesign.

The projection differences are enumerated in [`spec/artifacts/run-result/README.md`](../../spec/artifacts/run-result/README.md): the stdout document adds `artifactRoot`, and drops `noop` and the evidence `sha256` digests. Everything else is shared, and projection parity is verified per fixture by `run_result_fixtures.rs`.

## Case result

Case-level results are recorded in the run manifest's `tests[]` entries; there is no per-case `result.json` in the current layout. stdout and stderr are written as files and referenced by the manifest, never fully embedded in JSON.

## Coverage artifacts

Coverage artifacts belong to adapters and shims. The runner should preserve raw coverage artifact locations when adapters produce them.

The exact structure should remain adapter-defined until real coverage adapters exist.

## File-level errors

When the pre-execution validation phase finds files that cannot be read or parsed, the run manifest records them as `diagnostics[]` entries (`category: "parse"` for parse errors, `category: "internal"` for read errors) with `status: "error"` and `processExitCode: 2`, regardless of whether any cases were evaluated. The representative shape is pinned by `tests/fixtures/run_result/parse_error.snapshot.json`.

## Artifact write failures

Artifact write failures are runtime infrastructure errors.

Since artifacts are generated by default, inability to create the artifact directory or write required result files should not be silently ignored.

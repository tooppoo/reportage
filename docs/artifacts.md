# Artifacts

This document describes the current v0 artifact direction for reportage.

Artifacts are evidence produced by a reportage run. They allow humans, CI systems, adapters, and later post-processing tools to inspect what happened during E2E execution.

## Default behavior

Artifacts are generated by default for every run.

There is currently no `--no-artifacts` mode in the v0 direction. Artifact generation is treated as a normal part of reportage execution, not as an optional reporting feature.

## Stability

The artifact layout and JSON schema should be considered experimental during early v0 implementation.

The implementation should generate artifacts early so the real output can be inspected and adjusted before stabilizing the format.

Expected stabilization direction:

```text
v0.0.x: generate artifacts, schema experimental
v0.1.x: begin stabilizing result.json shape
v0.2.x: consider external post-processing and testography integration needs
```

## Default layout

The default artifact root is:

```text
.reportage/
  runs/
    <run-id>/
```

A typical run layout may be:

```text
.reportage/
  runs/
    <run-id>/
      result.json
      cases/
        <case-id>/
          result.json
          stdout.txt
          stderr.txt
          coverage/
          workspace/
```

`workspace/` is only expected when a workspace is preserved for debugging or when a later option explicitly requests preservation.

## Run result

`result.json` at the run root should summarize the full run.

Initial shape:

```json
{
  "schemaVersion": 1,
  "status": "failed",
  "cases": [
    {
      "id": "check-output-json",
      "name": "check output / json",
      "status": "failed",
      "resultPath": "cases/check-output-json/result.json"
    }
  ]
}
```

The exact schema is experimental in early v0.

## Case result

Each case should have a case-level result artifact.

stdout and stderr may be written as files and referenced by result JSON rather than fully embedded in JSON.

## Coverage artifacts

Coverage artifacts belong to adapters and shims. The runner should preserve raw coverage artifact locations when adapters produce them.

The exact structure should remain adapter-defined until real coverage adapters exist.

## File-level errors

When the pre-execution validation phase finds files that cannot be read or parsed, the run artifact records them under `file_errors`.

```json
{
  "result": "script_error",
  "cases": [],
  "file_errors": [
    {
      "source_path": "examples/broken.repor",
      "kind": "read_error",
      "message": "No such file or directory (os error 2)"
    },
    {
      "source_path": "examples/invalid.repor",
      "kind": "parse_error",
      "message": "parse error at line 3: ..."
    }
  ]
}
```

`kind` is either `read_error` (I/O or permission failure) or `parse_error` (syntax error).

`result` is `script_error` when any file-level errors exist, regardless of whether any cases were evaluated.

## Artifact write failures

Artifact write failures are runtime infrastructure errors.

Since artifacts are generated by default, inability to create the artifact directory or write required result files should not be silently ignored.

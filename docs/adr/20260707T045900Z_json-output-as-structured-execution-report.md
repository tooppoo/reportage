# JSON Output as a Structured Execution Report

- Status: Accepted
- Created: 2026-07-07T04:59:00Z

## Context

Reportage aims to be an e2e testing tool that AI, CI, and external tooling can depend on. That means `reportage run`'s result must be obtainable in a machine-readable form, not only as human-readable console output.

If JSON output is implemented as a stringification or simplified version of the human-readable output, several problems follow:

- The human-readable output and JSON output would have diverging sources of truth.
- The runner would accumulate per-format responsibility.
- Semantic information present in the human-readable output could silently go missing from JSON output.
- AI, CI, and external tooling would have no stable contract to depend on.
- The classification of parse error / semantic error / runtime error / assertion failure would be tied to display wording.

[`20260706T001018Z_separate-execution-report-from-output-rendering.md`](20260706T001018Z_separate-execution-report-from-output-rendering.md) (issue #76) restructured the runner so it produces a format-agnostic `ExecutionReport` rather than human-readable output directly, and issue #75 added a `render::json::JsonRenderer` that turns that `ExecutionReport` into a JSON document (`crates/reportage-cli/src/render/json.rs`). This ADR is the formal decision record for that renderer's contract, plus the diagnostic/failure model it exposes, deferred by #75's own commit message to issue #89.

Related issues: [#75](https://github.com/tooppoo/reportage/issues/75), [#76](https://github.com/tooppoo/reportage/issues/76), [#89](https://github.com/tooppoo/reportage/issues/89).

## Decision

`reportage run --format=json` produces a structured execution report derived from `ExecutionReport`, not a derivative of the human-readable output.

The runner (`evaluator`, `executor`) never produces JSON directly. It produces a display-format-independent `ExecutionReport`; the CLI / output layer resolves `--format=json` and selects `JsonRenderer`.

Human-readable output and JSON output are both generated by reading the same `ExecutionReport`.

JSON output must not fall below the semantic information present in ordinary human-readable output. "Semantic information" here means structured information needed to interpret the result — not ANSI color, box-drawing characters, whitespace, or wording.

JSON output must be able to represent at least:

- top-level `status`
- top-level `processExitCode`
- the parse / semantic / runtime / assertion failure classification
- a stable diagnostic code
- severity
- message
- source location/span, if available
- origin, if location is unavailable
- action id
- assertion id
- checkpoint / action reference
- expectation kind
- expected value
- actual value, or a reference to it
- artifact root
- captured stdout/stderr artifact reference, if captured output exists

### Diagnostic and failure model

JSON output treats error / failure / warning uniformly as a diagnostic model, but distinguishes assertion failure from parse/semantic/runtime error.

- `parse.*` — a source syntax or parse-domain validation failure.
- `semantic.*` — the script, normalized semantic model, or expectation definition is invalid, and evidence comparison cannot proceed. This also covers `ParseMissingAssertionBlock`, the one parse-domain code detected at case-evaluation time rather than by the parser itself (see `evaluator::evaluate_case`); its diagnostic `category` is still reported as `parse`, matching where the failure conceptually belongs, not where it happened to be detected.
- `runtime.*` — an action execution infrastructure failure (e.g. executable not found, spawn failure, timeout, artifact write failure). This category also covers `step.*` diagnostic codes (e.g. `step.write.target_exists`): a side-effecting step's own runtime failure is reported under the `runtime` category in this JSON contract, even though `docs/semantic-diagnostics.md` documents `step.*` as its own top-level code namespace distinct from `parse.*`/`semantic.*`/`assertion.*`. The JSON `category` enum groups by "what kind of failure is this for a JSON consumer to react to", not by code-namespace prefix.
- `assertion.*` — script and evidence acquisition succeeded, but the observed evidence does not satisfy the expectation.
- `internal` — failures that predate parsing entirely (currently only a file read error) and therefore fit none of the above namespaces. This is an addition beyond the four namespaces above, not a fifth failure classification that consumers need to reason about symmetrically with the other four.

An assertion failure included in `diagnostics[]` is distinguished from an error by `severity: "failure"` and `code` matching `assertion.*`, as opposed to `severity: "error"` for every other category.

A command's non-zero exit is not itself treated as a runtime error. Non-zero exit is observed evidence of an action execution result, compared against an `exit <code>` expectation; the comparison result is an assertion outcome (pass or `assertion.exit.mismatch`), never a `runtime.*` diagnostic on its own.

Top-level `status` precedence is:

```text
error > failed > passed
```

- `status: "error"` if any file-level error exists, or any test has a `ScriptError`/`RuntimeError` status.
- Otherwise `status: "failed"` if any test failed (an unmet assertion).
- Otherwise `status: "passed"`.

## Alternatives Considered

### Implement JSON output as a stringification of human-readable output

Rejected. Human-readable output is meant for people: wording, decoration, and layout can change. Deriving JSON from it would make the machine-readable contract unstable in exactly the ways it needs to be stable.

### Have the runner branch on output format directly

Rejected. If the runner directly handled `json` / human / future formats (`ndjson`, `junit`, `tap`, GitHub annotations, ...), output-format concerns would leak into execution logic. The `ExecutionReport` → renderer structure from #76 is preserved; `JsonRenderer` is one more `OutputRenderer` implementation, nothing more.

### Treat a non-zero process exit code as a runtime error

Rejected. Many scripts intentionally assert `exit <nonzero>`. Treating non-zero exit as `runtime.*` would misclassify an entirely expected, successfully-observed outcome as an infrastructure failure.

## Consequences

### Positive Consequences

- AI, CI, and external tooling can depend on structured JSON fields instead of human-readable wording.
- Because both renderers read the same `ExecutionReport`, a semantic-information gap between them is a detectable bug, not an accepted drift.
- The failure/diagnostic classification is explicit and testable independently of display text.

### Negative Consequences

- JSON output is a second contract surface (alongside human-readable output and the pre-existing `result.json` artifact) that must stay in sync with the internal model as it evolves.
- The `runtime` category intentionally spans two internal code namespaces (`runtime.*` and `step.*`); a reader of `docs/semantic-diagnostics.md` alone, without this ADR, could expect a `step` JSON category that does not exist.

### Neutral Consequences

- `result.json` (the pre-existing artifact written by `reportage_core::artifact::ArtifactWriter`) uses a different, older, snake_case vocabulary (see `docs/exit-codes.md`) and is unaffected by this ADR. See [`20260707T050100Z_json-output-schema-and-validation-policy.md`](20260707T050100Z_json-output-schema-and-validation-policy.md) for how the two are kept distinct.

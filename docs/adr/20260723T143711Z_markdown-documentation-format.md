# Markdown Documentation Format

- Status: Accepted
- Created: 2026-07-23T14:37:11Z

## Context

Issue #171 adds `--format markdown` to `reportage docs`, generating a single-file Markdown document from the Documentation Catalog introduced in #170, plus a format-shared `--title` option.
The Catalog, glob resolution, source loading, ordering, fallback, and output policies stay unchanged; this ADR records only the durable serialization decisions for the Markdown format and the document title.

Markdown output raises questions that plain text did not: whether metadata must be escaped for Markdown, how table-of-contents links stay stable across Markdown implementations, and how case sources survive being embedded in fenced code blocks.
These decisions shape the user-visible contract and are likely to be questioned again when an HTML format or multi-file layouts arrive, so they are recorded here.

## Decision

### Metadata is mapped verbatim, without Markdown escaping or sanitization

The document title, group names, file titles, case titles, source paths, and descriptions are inserted into the generated Markdown directly.
The renderer never escapes, sanitizes, trims, or dedents them, and never parses or restructures them as a Markdown AST.
If a value contains Markdown syntax, raw HTML, or newlines that break the rendered layout or table of contents, that is the input's responsibility.
Sanitizing Markdown generated from untrusted `.repor` sources is the responsibility of the downstream publisher or rendering environment, not of `reportage docs`.

### Explicit anchors from Catalog structure indices, not implicit slugs

The table of contents links through explicit `<a id="...">` anchors generated from the 1-based Catalog structure indices (`group-1`, `file-1-2`, `case-1-2-3`), optionally suffixed with a normalized readability slug.
Uniqueness is guaranteed by the structure indices alone; the slug is a readability aid and is omitted when normalization leaves nothing.
Anchor IDs contain only renderer-generated ASCII lowercase letters, digits, and hyphens, so raw metadata never lands inside an HTML attribute.

### Case sources are wrapped in variable-length fences and never modified

Each case source is embedded in a fenced code block with the `reportage` language identifier.
The fence is one backtick longer than the longest backtick run in the exact source (minimum three), so a source containing backtick sequences cannot terminate the block early.
Inside the fence, the renderer normalizes CRLF to LF and may add one structural LF when the source has no final newline (so the closing fence sits on its own line); beyond that, source content is never dropped or replaced.

### The document title is a format-shared render option, not a Catalog property

`--title` flows to every format through a document-level render option next to the Catalog in the renderer interface.
It is never stored in the source-level model or in the Catalog, so it cannot affect Catalog ordering, fallbacks, or anchor IDs, and the same Catalog renders identically apart from the title line.

## Alternatives Considered

### Escaping or sanitizing metadata for Markdown

Escaping would require choosing one Markdown dialect's escaping rules and re-encoding every metadata value, and sanitizing raw HTML would require an HTML policy inside a documentation generator.
Both make the output depend on parsing decisions the renderer cannot get right for every downstream implementation, and both silently alter what the author wrote.
Rejected: authors see their metadata verbatim, and trust decisions stay with the publisher who knows the rendering environment.

### Relying on implicit heading slugs for the table of contents

GitHub, GitLab, and other renderers derive heading anchors with different, undocumented, and mutually incompatible slug rules, and none of them guarantees stability for duplicate or non-ASCII headings.
Rejected: explicit anchors generated by the renderer are the only way to keep intra-document links deterministic and implementation-independent.

### A fixed three-backtick fence

A case source containing a three-backtick run would terminate the block early and corrupt the document.
Rejected in favor of computing the fence length from the source content.

### Storing the title in the Catalog

Threading the title through the Catalog would blur the boundary between source-derived properties and invocation options, and would force Catalog tests to care about a CLI flag.
Rejected: the render-option boundary keeps the Catalog a pure function of the sources.

## Consequences

### Positive Consequences

- Anchors are deterministic, collision-free, and portable across Markdown implementations.
- Case sources round-trip losslessly (modulo LF normalization) regardless of embedded backticks or missing final newlines.
- Renderers stay independent of parser types and CLI concerns; adding an HTML format reuses the same Catalog and render options.

### Negative Consequences

- Metadata containing Markdown syntax can break the generated layout; the tool offers no protection for untrusted sources, and publishers must sanitize downstream.
- Explicit `<a id>` anchors add HTML noise to the raw Markdown text.

### Neutral Consequences

- The slug part of an anchor changes when a title changes, so deep links into published documentation are stable only while titles are stable; the index part alone changes when ordering changes.

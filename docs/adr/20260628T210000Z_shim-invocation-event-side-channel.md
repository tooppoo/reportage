# Shim Invocation Event Side Channel

- Status: Accepted
- Created: 2026-06-28T21:00:00Z

## Context

Runner-owned PATH overlay shims may shadow ambient commands so that `$` actions
in `.repor` scripts invoke the runner's wrapper rather than any command found
on the ambient `PATH`. This is the intended model (see ADR
`20260628T061500Z_path-overlay-shims-for-command-resolution`), but it can make
test failures hard to diagnose when the relationship between the command name
in the script, the shim that was resolved, and the target executable behind the
shim is not visible in diagnostics or artifacts.

The runner needs a way to observe which shim intercepted a command and what
executable invocation the shim delegated to, without having to parse the action
command text.

## Decision

### Command text is not parsed to infer shim usage

reportage must not parse action command text to determine whether a shim was
involved in executing it.

Action commands are passed verbatim to `sh -c`. Command resolution — which
binary on `PATH` actually ran — is performed by the POSIX shell and the runtime
environment. Attempting to infer shim usage from arbitrary action command text
would require reportage to partially reimplement shell parsing and `PATH`
lookup semantics. This is not the intended model, would be fragile, and would
not generalise to all shell syntax (pipelines, subshells, variable expansion,
aliases, etc.).

### Shims report invocations through a runner-provided side channel

A protocol-compliant shim — one generated by reportage or written to the same
contract — writes a JSON event file to a runner-provided directory before
exec-ing its target invocation. The runner reads these event files after the
action completes and attaches them to the corresponding action result.

This keeps command resolution in the shell/runtime environment and limits
reportage's responsibility to providing the side channel and reading event
files from it.

### Action-scoped event directory

The side channel is an action-scoped event directory. Before executing an
action, the runner creates a fresh temporary directory for that action and
exposes it through the `REPORTAGE_SHIM_EVENT_DIR` environment variable. Any
protocol-compliant shim invoked during the action writes its event file into
that directory. After the action completes, the runner reads that directory and
attaches the collected events to that action's result.

The action-scoped directory provides two guarantees:
1. **Action attribution**: events are attributed to the action that produced them,
   not to any other action.
2. **Stale-event isolation**: events from one action cannot be attached to a
   later action, because each action gets a fresh directory.

The directory does not prevent all file-name collisions within a single action.
Protocol-compliant shims use the shell PID (`$$`) as the event file name to
avoid collision when multiple shims are invoked within the same action.

### Event file format

Each shim invocation writes one event file in JSON format:

```json
{
  "schema_version": 1,
  "event": "shim_invoked",
  "command_name": "<the command name this shim shadows>",
  "shim_path": "<absolute path to the wrapper script>",
  "target": {
    "program": "<absolute path to the target executable>",
    "args": ["<fixed-arg-1>", ...]
  },
  "forwards_caller_args": true
}
```

The target is modeled as an executable invocation (program + fixed args) rather
than a binary-only path. This allows targets such as `ruby tool.rb` to be
represented without changing the model.

### Warning-and-continue policy for event write failures

A runner-generated shim that fails to write its invocation event emits an
explicit stderr diagnostic and continues exec-ing to its target invocation:

```
reportage shim warning: failed to write shim invocation event: <path>
```

The shim must not fail silently and must not abort the delegated invocation.
The user's test behavior is preserved; the diagnostic on stderr leaves evidence
that shim metadata capture failed.

Runner-generated shim stderr diagnostics are deliberately observable stderr.
They are not automatically filtered out from stdout/stderr assertions. If a
`stderr empty` assertion is in use, a shim write failure will make it fail.
This is intentional: the user must see that something unusual happened.

### Shim invocation metadata is observed evidence, not a complete PATH trace

Shim invocation metadata records what the runner observed through the event
side channel. It does not prove that no shim was used when metadata is absent.
A non-protocol-compliant shim, an ambient command, or a direct binary invocation
produces no event and leaves no trace in the action result. The runner records
what it received; absence means "not observed via the protocol", not "no shim
was involved".

### The runner owns final artifact integration

Shims write event data to the side channel. The runner reads it and decides how
to integrate the data into the artifact. The shim must not mutate the final
result artifact directly.

## Alternatives Considered

### Parse action command text to detect shim usage

This would avoid the side-channel protocol entirely: the runner could inspect
the command string, resolve it against the known PATH prefixes, and determine
whether a shim would be hit.

This is rejected because:
- It requires partial shell parsing. The command text may use pipelines,
  subshells, aliases, variable-expanded names, or other constructs. A partial
  parser would be fragile and incomplete.
- It would push command-resolution logic into the runner, duplicating what the
  shell already does.
- It would not generalise to commands resolved indirectly (e.g., a script that
  invokes a shim from within a subshell).

### Emit shim metadata from the runner before/after the action

The runner knows which shims it materialized and which PATH prefix directories
it prepended. It could speculatively record which commands might have been
intercepted based on the command text and the known shims.

This is rejected for the same reasons as parsing the command text. It would
produce speculative, not observed, metadata. A command may invoke a shim zero
or many times within a single `$` action; the runner cannot know without
observing actual shell execution.

### Per-run shared event file

A single event file shared across all actions in a run, with each event record
stamped with an action identifier.

This is rejected because action attribution becomes complex: the runner must
match events to actions reliably, and concurrent future execution would make
this harder. An action-scoped directory is simpler and attribution is automatic.

### Post-execution event write by the shim

Having the shim write the event file after the target returns, rather than
before exec-ing it.

This is rejected because protocol-compliant shims use `exec` to replace the
shim process with the target process. Once `exec` is called, the shim process
no longer exists and cannot write post-execution data. All event writing must
happen before the `exec`.

## Consequences

### Positive Consequences

- The runner does not need to parse or understand shell syntax.
- Command resolution remains entirely the shell's responsibility.
- Action results carry observed shim invocation evidence when a
  protocol-compliant shim is involved.
- Failure diagnostics include the shim path and target invocation, making PATH
  overlay failures easier to diagnose.
- The event model generalises to future third-party shims that implement the
  same contract.

### Negative Consequences

- Non-protocol-compliant shims produce no event. Their invocation is
  indistinguishable from a direct ambient command.
- The side channel adds a temp-directory creation and read to every action
  execution, even when no shims are involved.
- Shim write failures appear on stderr and can cause `stderr empty` assertions
  to fail, which may surprise users until they investigate.

### Neutral Consequences

- The `REPORTAGE_SHIM_EVENT_DIR` variable is an internal protocol detail. It
  is set by the runner and read by the shim. User scripts must not rely on it.
- The `$$` (shell PID) naming scheme for event files works well for sequential
  shim invocations within a single action. PID reuse within the same action is
  not possible in practice because two concurrent processes cannot share a PID.
- Malformed event files produce runner-level warnings in the action result, not
  hard errors, so a bad event file never aborts the run.

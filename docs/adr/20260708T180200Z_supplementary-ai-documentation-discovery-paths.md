
# Supplementary AI Documentation Discovery Paths for v0

- Status: Proposed
- Created: 2026-07-08T18:02:00Z

## Context

[Tag-Based GitHub URLs and `reportage docs` as the v0 AI Documentation Discovery Path](20260708T180000Z_ai-documentation-discovery-core-path.md) establishes `reportage docs` as the v0 core discovery path. But an AI will not always run `reportage docs` on its own initiative. When an AI is fixing a failure, being able to reach the relevant documentation directly from JSON output or diagnostics could make the fix loop more reliable.

At the same time, `llms.txt`, editor/agent skills, an MCP server, and a dedicated Web site all carry tool-specific integration cost, which makes them too heavy to be v0's core discovery path.

This is the third of three ADRs recorded from issue #136. It records how the remaining candidate paths are prioritized and bounded for v0.

Related issues: #136, #147, #148.

## Decision

The following paths are treated as supplementary, not core, discovery paths for v0, split into two priority tiers.

P1:

- `reportage init --ai`, which generates AI-facing reference file(s) inside the user's own project;
- documentation URLs surfaced from JSON output / diagnostics;
- package/registry metadata.

P2:

- `/llms.txt` and `/llms-full.txt`;
- a Claude Code skill or plugin;
- an MCP server;
- GitHub Pages or a dedicated Web site;
- LSP/IDE integration.

Files generated by `reportage init --ai` inside a user's project are pointers, not an authoritative copy of the documentation. Initial exploration should prefer a reportage-specific file such as `.reportage/ai.md` or `.reportage/docs.md`. Automatically writing to `AGENTS.md`, `CLAUDE.md`, or `.github/copilot-instructions.md` must be treated cautiously, since those files may already exist and may be read by unrelated tools.

Where JSON output or diagnostics surface a documentation URL, that URL must be built using the same tag-based rule as `reportage docs` (see [the core discovery path ADR](20260708T180000Z_ai-documentation-discovery-core-path.md)).

Package metadata for v0 points at the repository, not at a documentation site:

```toml
[package]
repository = "https://github.com/tooppoo/reportage"
```

A `documentation` field pointing at a `latest`-style URL, such as `documentation = "https://reportage.dev/docs/latest/"`, is not adopted in v0, for the same reason `reportage docs` does not support a `latest` URL (see [the core discovery path ADR](20260708T180000Z_ai-documentation-discovery-core-path.md)).

## Alternatives Considered

### Treat `llms.txt`, a skill/plugin, or an MCP server as a v0 core path

Rejected. Each of these paths only benefits a subset of AI tooling (e.g. a Claude Code skill has no effect on ChatGPT, Cursor, or Copilot), and each carries its own integration and maintenance cost. `reportage docs` remains the one path every AI tool can reach through an ordinary shell command.

### Point package `documentation` metadata at a `latest` docs URL

Rejected, for the same reason `reportage docs` excludes `latest`: a `documentation` field that always points at the newest docs would drift from whatever reportage version a given user actually has installed.

## Consequences

### Positive Consequences

- The core path stays concentrated on `reportage docs`, while supplementary paths can be added incrementally.
- Diagnostics and JSON output give an AI a way to reach documentation even when it does not run `reportage docs` directly.
- Fix loops that start from a failing diagnostic can reach the relevant documentation without an extra discovery step.
- No single AI tool's ecosystem is treated as privileged.

### Negative Consequences

- As `reportage init --ai` generates more files, the user's project accumulates more generated files to maintain.
- Adding documentation URLs to JSON output requires tracking output size and schema compatibility.
- Deferring the skill/MCP/Web site paths to P2 means tool-specific experiences remain limited in the near term.
